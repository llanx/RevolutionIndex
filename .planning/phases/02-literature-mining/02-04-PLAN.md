  <action>
Create the ranked variable catalog by synthesizing Variables Discovered tables from all 6 domain review documents. Follow the catalog structure from 02-RESEARCH.md (Pattern 2: Variable Catalog Entry).

**Step 1: Extract and deduplicate variables**

Read all 6 domain review documents and extract every variable from their Variables Discovered tables. Deduplicate at the concept level per 02-RESEARCH.md Open Question #6: "Income inequality" is one catalog entry with multiple proxy rows (Gini, top 1%, labor share), not 10 separate entries. Same concept appearing under different names across domains (e.g., "relative deprivation" in Domain 1, "perceived loss" in Domain 4, "economic grievance" in Domain 3) should be unified into one entry with cross-domain citations.

**Step 2: Apply hybrid rating system (locked decision)**

Rate each variable using the locked criteria from 02-CONTEXT.md:
- **Strong**: Statistically significant coefficients in 2+ independent quantitative studies
- **Moderate**: Appears in 1 quantitative model OR strong qualitative consensus across 3+ studies
- **Weak**: Theoretically motivated but limited testing, or single qualitative mention

Add **Contested** marker where evidence conflicts exist (per locked decision).

Apply the **measurability filter**: Only catalog variables that have at least one known measurement approach or proxy. Theoretically important but unmeasurable variables are excluded entirely (not rated "Weak" -- excluded).

**Step 3: Cross-reference against MCP federal data APIs**

For each variable, perform a quick (~30 second) availability check using the US Government Open Data MCP:
- Use `fred_search`, `bls_search_series`, `census_search_variables`, etc. to spot-check whether a known federal data source exists
- Tag each variable: `fed-data` (known federal API source), `other-data` (known non-federal source like WID, V-Dem, Pew), `unknown` (no source identified yet)
- This is a QUICK cross-reference, NOT exhaustive data sourcing. Do NOT retrieve actual data, document series IDs, or verify update frequencies. That is Phase 3's job.
- Refer to the MCP Federal API Coverage Map in 02-RESEARCH.md for guidance on which APIs to check for which variable categories.

**Step 4: Write the catalog document**

Structure the document as:

```
# Variable Catalog: Revolution and Political Instability Predictors

## Methodology
[Brief description of rating system, measurability filter, data availability tagging]

## Summary Table
| # | Variable | Domain(s) | Rating | Contested? | Data Availability | Key Studies |
[One row per variable, sortable by rating]

## Detailed Entries
### [Variable Name]
[Full Pattern 2 entry for each variable]
```

**Catalog at the concept level with measurement sub-entries:** Each variable entry in the detailed section lists ALL known measurement approaches/proxies from the citing studies, per the locked "proxy capture" decision. For example:

```
### Income Inequality
**Rating:** Strong
**Domain(s):** Economic Preconditions, Revolution Prediction, Historical Case Studies
**Data Availability:** `fed-data`

**Evidence:**
| Study | Proxy Used | Finding | Type |
| Muller (1995) | Gini coefficient | Higher inequality -> more political violence | quant |
| Alesina & Perotti (1996) | Income share ratios | Inequality destabilizes middle-income countries | quant |
...

**Measurement Approaches:**
- Gini coefficient: Census ACS, BEA
- Top 1% income share: WID (other-data)
- Labor share of GDP: FRED W270RE1A156NBEA (fed-data)
- Top 10% wealth share: Fed Survey of Consumer Finances
```

**Target:** 40-60 variables total. If significantly more emerge, tighten the measurability filter. If significantly fewer, review whether domain reviews adequately captured variables.

**Anti-patterns to avoid:**
- Do NOT include variables without at least one known measurement approach (measurability filter)
- Do NOT rate more than 50% of variables as "Weak" (sign of insufficient discrimination)
- Do NOT catalog frameworks as variables (e.g., "Structural-demographic theory" is a framework, not a variable)
- Do NOT include FRED series IDs or API endpoints (that is Phase 3 scope)
- Do NOT spend more than ~30 seconds per variable on the MCP availability check
